# DAG任务调度系统 竞品对比与架构优化方案

> 作为重度使用DAG任务的企业级用户，从应用程序架构师视角给出竞品分析、架构评估和优化计划。

---

## 一、主流DAG任务调度系统竞品对比

| 产品 | 架构模式 | 优点 | 缺点 | dag-task对比优势 | dag-task对比不足 |
|------|---------|------|------|-----------------|-----------------|
| **Apache Airflow** | 中心化调度 + Worker执行 | 功能丰富，UI强大，生态完善 | 过重，资源消耗大，启动慢，分布式扩展复杂 | 轻量级，模块化设计，启动快，资源占用小 | 功能相对基础，生态不完善 |
| **Apache DolphinScheduler** | 去中心化多Master架构 | 高可用，可视化，支持大规模任务 | 架构复杂，学习曲线陡，运维成本高 | 架构简单清晰，运维成本低，易于理解 | 高可用方案尚未完善 |
| **Azkaban** | 中心化 + 多Executor | 简单易用，适合批处理 | 不支持动态DAG，界面老旧 | 支持动态DAG，更好的依赖处理 | 大规模集群支持不足 |
| **Prefect 2.0** | DAG as Code + API-first | 现代化设计，流处理友好 | 复杂部署，企业版闭源 | 完全开源，部署简单 | 云原生集成不足 |
| **Dagster** | 资产导向编程 | 数据 lineage 支持好 | 概念复杂，学习曲线陡 | 更直接的DAG模型，易于理解 | 数据 lineage 特性缺失 |
| **dag-task (当前项目)** | **Server + 多Agent 分离架构** | SQL-first设计，模块化，轻量 | 正在发展中 | - | - |

## 二、当前dag-task架构优势分析

### 已有的优良设计：

1. **清晰的分层架构**
   - `dag-si` 纯领域模型层，无实现依赖，符合整洁架构原则
   - 核心逻辑与HTTP服务分离，易于测试和替换

2. **正确的分离关注点**
   - 调度中心(Scheduler)只负责调度和状态管理
   - 执行Agent只负责任务执行，职责单一
   - 数据库驱动设计避免了内存一致性问题

3. **合理的并发模型(Agent侧)**
   - 三队列设计(pending/running/finished)
   - 有界队列防止OOM
   - 容量控制(返回429)保护Agent不被压垮

4. **健壮的错误处理**
   - 死信队列持久化失败上报，自动重试
   - 优雅关闭支持，不丢弃正在执行的任务

## 三、架构优化建议（按优先级排序）

### P0 核心能力完善（当前最需要）

#### 1. **任务失败自动重试**
**现状**: 没有内置重试机制，需要用户手动处理

**建议方案**:
- 在任务记录中增加重试次数字段 `retry_count` 和 `max_retry`
- 失败后自动增加重试计数，如果未超过最大值，重新放入就绪队列
- 支持退避策略 (fixed / exponential backoff)
- 重试间隔可配置

#### 2. **任务超时处理**
**现状**: 缺少强制超时中断

**建议方案**:
- 任务级别的超时配置 `timeout_seconds`
- Scheduler定时扫描长时间运行的任务
- 超时后自动调用kill并标记为ERROR

#### 3. **事件通知系统**
**现状**: 没有内置通知机制

**建议方案**:
- 定义回调接口 `NotificationSender`
- 支持邮件/企业微信/钉钉/Slack通知
- 可配置：失败通知/成功通知/完成通知
- 支持模板配置

---

### P1 架构健壮性提升

#### 4. **Scheduler高可用支持**
**现状**: 单点设计

**建议方案**:
- 基于数据库乐观锁实现Leader选举
- 多个Scheduler节点同时启动，只有一个Leader在跑调度
- Leader挂掉后，其他节点自动选举新Leader
- 不需要ZooKeeper，减少外部依赖

#### 5. **Agent负载均衡与分组**
**现状**: 当前只有简单的负载均衡，缺少分组

**建议方案**:
- Agent增加标签(label/group)支持
- 任务可以指定要运行在哪个group/label的Agent
- 支持任务亲和性调度
- 便于隔离不同类型的工作负载

#### 6. **调度策略增强**
**现状**: 只支持就绪即调度

**建议方案**:
- 支持定时任务 (cron表达式)
- 支持依赖任务成功完成才触发 (已有，保持)
- 支持手动触发
- 支持暂停/恢复 (已有hold/free，完善状态转移)

#### 7. **历史任务清理**
**现状**: 数据一直累积

**建议方案**:
- 支持按时间/条数清理历史完成任务
- 可配置保留策略 (保留N天/保留N条)
- 支持清理到归档表，不删除原始数据

---

### P2 企业级特性增强

#### 8. **权限管理**
**现状**: 没有权限控制

**建议方案**:
- 基于RBAC的权限模型
- 用户/角色/权限管理
- 工作流级别的访问控制

#### 9. **审计日志**
**现状**: 只有普通应用日志

**建议方案**:
- 独立的审计日志表
- 记录所有操作：谁在什么时候对什么任务做了什么操作
- 支持查询审计日志

#### 10. **工作流版本管理**
**现状**: 没有版本控制

**建议方案**:
- DAG定义支持版本号
- 可以回滚到历史版本
- 记录每个版本的修改人/修改时间

#### 11. **条件分支支持**
**现状**: 只支持静态DAG

**建议方案**:
- 支持运行时条件判断 (if/else)
- 根据上游任务结果选择不同执行路径
- 保留当前静态DAG的简单性，可选启用

#### 12. **子工作流支持**
**现状**: 不支持嵌套

**建议方案**:
- 允许一个DAG节点引用另一个完整DAG
- 子DAG完成后父DAG继续执行
- 降低大型DAG复杂度，提高可复用性

## 四、性能优化建议

### 1. **数据库查询优化**
- 当前已经是SQL-first设计，很好
- 建议添加合适的联合索引：
  - `(status, order_key)` - 就绪任务查询
  - `(agent_id, status)` - Agent任务查询
  - `(create_time)` - 历史清理

### 2. **DAG缓存优化**
- 当前已经用Caffeine缓存DAG定义，很好
- 建议增加缓存失效策略：DAG更新时立即失效

### 3. **结果上报批处理**
- 当前已经支持批量上报，很好
- 建议：当结果队列满时，持久化到死信，不丢弃，很好保持这个设计

## 五、部署架构演进建议

### 当前部署：
```
[Web UI] <-HTTP-> [Scheduler Server] <-HTTP-> [Multiple Agents]
```

### 建议企业级部署：
```
                   ┌─────────────┐
                   │   LB/Nginx   │
                   └──────┬──────┘
                          │
           ┌───────────────┼───────────────┐
           │               │               │
      ┌────▼─────┐   ┌────▼─────┐   ┌────▼─────┐
      │Scheduler│   │Scheduler│   │Scheduler│
      │  Node 1 │   │  Node 2 │   │  Node 3 │
      └────┬────┘   └────┬────┘   └────┬────┘
           │         └──────┬──────┘      │
           │                │              │
           └────────┼───────┘──────────────┘
                    │
              ┌─────▼─────┐
              │ PostgreSQL │
              └─────┬─────┘
                    │
           ┌────────▼─────────┐
  ┌────────▼─────────┐   ┌────────▼─────────┐
  │   Agent Node 1    │   │   Agent Node 2    │  ...
  │   (tag=batch)     │   │  (tag=realtime)  │
  └───────────────────┘   └───────────────────┘
```

**关键改进**:
- 多个Scheduler节点通过数据库Leader选举实现高可用
- Agent按业务标签分组，隔离不同类型工作负载
- 仍然只需要一个PostgreSQL，不需要额外组件

## 七、总结

当前**dag-task**的架构设计是**优良**的：

- ✅ 模块化清晰，领域模型分离
- ✅ SQL-first设计避免了很多分布式一致性问题
- ✅ 轻量级，没有过重的外部依赖
- ✅ 三队列并发模型在Agent侧设计合理

只需要按上面的计划逐步增加企业级特性，就能成为一个非常好用的轻量级企业级DAG调度系统。

**核心设计哲学保持不变**: *简单、清晰、够用*，这比过度设计的大型系统更容易维护和运维。
